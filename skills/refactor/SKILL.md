# Vibe Refactor Workflow

코드 구조 개선을 위한 리팩터링 경로: Phase 1 → Phase 3 → Phase 5 → Phase 6 → Phase 7

> **핵심 원칙: 기존 테스트가 깨지지 않는다.**
> Refactor 모드에서는 Iron Law("테스트 없는 코드 삭제")가 적용되지 않는다. 이미 테스트가 있는 기존 코드를 다루기 때문이다.

## Phase 시작 공통 절차

**모든 Phase 진입 시 반드시 실행:**

1. `.claude/workflow-config.yaml` 읽기 (없으면 기본값 사용)
2. `.claude/workflows/` 에서 현재 워크플로우 status 파일 읽기 (있으면)
3. 사용 가능한 스킬 목록을 조회하고, 현재 Phase의 작업과 1%라도 관련된 스킬이 있으면 반드시 호출한다. 스킬을 쓸지 말지 고민되면, 쓴다.
4. config 값에 따라 Phase 진행

## engagement_level

config의 `engagement_level`에 따라 유저 확인 빈도를 조절한다.

- **high**: 설계 승인 + 리팩터링 중간 확인 + 리뷰 결과 + 커밋 확인
- **balanced**: 설계 승인 + 리뷰 결과 + 커밋 확인
- **low**: 설계 승인 + 커밋 확인만

---

## Phase 1: 설계 (리팩터링용)

리팩터링 범위와 목표를 정의한다.

<HARD-GATE>
설계가 승인되기 전에는 리팩터링을 시작하지 않는다.
</HARD-GATE>

### 1.1 범위 확인

1. **프로젝트 컨텍스트 탐색** — 현재 코드 구조, 문제점 파악
2. **기능 변경 여부 확인** — "이 리팩터링으로 기능이 변경되나요?"
   - 기능 변경이 포함되면 → `/vibe:feature`로 전환 권유
   - 구조 변경만이면 → 계속 진행
3. **리팩터링 목표 정의**
   - 무엇을 개선하는가 (파일 분리, 중복 제거, 패턴 변경 등)
   - 완료 기준은 무엇인가

### 1.2 근거 기반 서치

리팩터링 패턴이 적절한지 검증:

| 소스 | 도구 | 용도 |
|------|------|------|
| 공식 문서 | context7 | 프레임워크별 권장 패턴 |
| 사례 | WebSearch | 리팩터링 베스트 프랙티스 |

**원칙: 재료는 검증된 것만, 요리(조합/판단)는 적극적으로**

### 1.3 설계 제시

리팩터링 계획을 유저에게 제시:

```
## 리팩터링 설계

**목표:** [개선 목표]
**범위:** [변경 대상 파일/모듈]
**방법:** [구체적 리팩터링 기법]
**영향:** [변경으로 인한 영향 범위]
**기능 변경 없음 확인:** ✅

진행할까요? [승인 / 수정 요청]
```

### 1.4 규모 판단

- **Small** (단일 파일 내부 정리): 설계 요약만 → Phase 3 직행
- **Standard** (여러 파일 구조 변경): 전체 프로세스
- **Large** (대규모 아키텍처 변경): 단계별 분리 후 진행

### Status 업데이트

Phase 1 완료 시: status에 phase=1 completed, 리팩터링 목표 기록. log에 설계 요약 append.

---

## Phase 3: 리팩터링

기존 테스트를 안전망으로 삼아 코드 구조를 개선한다.

### 3.1 사전 확인 (필수)

리팩터링 시작 전 반드시:

1. **기존 테스트 전부 실행** → 전부 통과 확인
2. **현재 커버리지 측정** → 기준선 기록
3. 통과하지 않는 테스트가 있으면 → 먼저 수정 후 리팩터링 시작

```
이 단계를 건너뛰면 리팩터링으로 인한 문제인지 기존 문제인지 구분할 수 없다.
```

### 3.2 리팩터링 사이클

```
리팩터링 한 단위 수행
  → 테스트 전부 실행 → 통과 확인
  → 다음 단위
  → 반복
```

한 단위가 클 때: 더 작은 단위로 쪼갠다. 각 단위 후 테스트가 통과해야 다음으로 진행.

### 3.3 리팩터링 체크 항목

- 긴 함수 (>50줄) → 분리
- 깊은 중첩 (>4레벨) → 평탄화
- 뮤테이션 → 이뮤터블 패턴 전환
- 매직 넘버 → 상수 추출
- 중복 코드 (3회 이상) → 추출
- 하나의 파일이 너무 클 때 (>800줄) → 책임 분리
- 1-2회 중복은 허용 (과잉 추상화 방지)

### 3.4 테스트 추가

리팩터링 과정에서 테스트가 부족한 영역을 발견하면:

1. 리팩터링 전에 해당 영역 테스트 추가
2. 테스트 통과 확인
3. 그 다음 리팩터링

### 3.5 커버리지 게이트

리팩터링 완료 후: config의 `defaults.coverage_threshold` 이상인지 확인. 리팩터링으로 커버리지가 떨어지면 안 된다.

### Status 업데이트

리팩터링 완료 시: status에 phase=3 completed, 변경 요약 기록. log에 리팩터링 내역 append.

---

## Phase 5: 코드 리뷰

리팩터링된 코드를 리뷰한다. Feature 모드와 동일한 구조.

### 5.1 내부 리뷰

config의 `defaults.review_mode`에 따라:
- **parallel**: 3개 에이전트 (품질/보안/테스트) 병렬 실행
- **single**: 단일 code-reviewer
- **Small**: review_mode 무시, 단일 리뷰어

각 에이전트에는 git diff + 리팩터링 설계만 전달.

리뷰 관점 추가: **기존 기능이 변경되지 않았는지** 확인.

### 5.2 리뷰 결과

```markdown
## 코드 리뷰 결과
### Critical (반드시 수정)
### Important (수정 추천)
### Minor (유저 판단)
```

유저 확인 → 이슈 처리.

### 5.3 Codex 외부 리뷰

config의 `defaults.codex_review`가 true일 때만. Small이면 생략.

```
/codex:review → 리팩터링 품질 검증
```

### Status 업데이트

Phase 5 완료 시: status에 phase=5 completed. log에 리뷰 결과 append.

---

## Phase 6: 검증

리팩터링 후 기존 기능이 정상 동작하는지 검증한다.

```
"테스트가 통과할 거야" → 금지. 실행 결과를 보여줘야 한다.
```

### 검증 레벨

- **Standard/Large**: FULL — 전체 테스트 + 빌드 + 린트 + 타입체크 + 커버리지
- **Small**: QUICK — 빌드 + 관련 테스트

### 5단계 게이트

IDENTIFY → RUN → READ → VERIFY → CLAIM. 실제 출력에 근거.

### 핵심 확인

1. **모든 기존 테스트 통과** — 하나라도 실패하면 리팩터링 롤백 검토
2. **커버리지 유지 또는 향상** — 리팩터링 전 대비 떨어지면 안 됨
3. **빌드 성공**

### 실패 시

테스트 실패 → 리팩터링이 기능을 깨뜨린 것. 해당 변경 되돌리고 재시도.

### Status 업데이트

Phase 6 완료 시: status에 phase=6 completed. log에 검증 결과 append.

---

## Phase 7: 마무리

검증을 통과한 리팩터링을 커밋하고 통합한다.

### 7.1 작업 완료 요약

```markdown
## 리팩터링 완료 요약
**목표:** [달성한 개선 목표]
**변경 내역:** 변경 파일 N개
**기능 변경:** 없음 ✅
**커버리지:** 리팩터링 전 N% → 후 N%
**검증:** 전체 통과
```

### 7.2 커밋

diff 요약 + 커밋 메시지 초안 제시 → 유저 승인 → 커밋. 메시지 형식: `refactor: <description>`

### 7.3 통합 방식

```
(1) 로컬 머지
(2) PR 생성
(3) 브랜치 유지
(4) 폐기
```

### 7.4 회고

```
"이번 리팩터링에서:
 - 범위가 적절했나요?
 - 테스트가 충분했나요?"
```

→ log에 회고 append → 필요 시 config 반영

### 7.5 정리

- status/log → `workflow-archive/`로 이동
- /clear 권유

---

## Status 관리

### 워크플로우 시작 시

`.claude/workflows/refactor-<target>.yaml` 과 `refactor-<target>-log.md` 생성.

### 업데이트 타이밍

| 이벤트 | status yaml | log md |
|--------|-------------|--------|
| Phase 전환 | phase 업데이트 | 완료 요약 append |
| 리팩터링 단위 완료 | progress 업데이트 | - |
| 커밋 | last_updated 갱신 | 커밋 메시지 append |
| 테스트 실패 | notes에 추가 | 상세 기록 append |

### 완료 시

`.claude/workflow-archive/YYYY-MM-DD-refactor-<target>.*` 로 이동.
