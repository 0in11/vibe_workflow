# Vibe Feature Workflow

새 기능을 구현하는 전체 파이프라인: Phase 1→2→3→(4)→5→6→7

## Phase 시작 공통 절차

**모든 Phase 진입 시 반드시 실행:**

1. `.claude/workflow-config.yaml` 읽기 (없으면 기본값 사용)
2. `.claude/workflows/` 에서 현재 워크플로우 status 파일 읽기 (있으면 이어서 진행)
3. 사용 가능한 스킬 목록을 조회하고, 현재 Phase의 작업과 1%라도 관련된 스킬이 있으면 반드시 호출한다. 스킬을 쓸지 말지 고민되면, 쓴다.
4. config 값에 따라 Phase 진행

**세션 재개 시:**
- `.claude/workflows/` 에 진행 중인 워크플로우가 있으면 목록을 제시하고 유저에게 선택받는다.

## engagement_level

config의 `engagement_level`에 따라 유저 확인 빈도를 조절한다.

- **high**: 모든 Phase 전환, 매 커밋, 리뷰 결과, 디버깅 보고, 검증 결과 확인 (~10회)
- **balanced** (기본): Phase 전환, 리뷰 결과, 최종 커밋만 확인. Phase 내부는 자동 진행. 이슈 시에만 중간 보고 (~4회)
- **low**: Phase 전환 시에만 확인. 나머지 자동 (~2회)

## Phase 1: 브레인스토밍

아이디어를 협업적 대화를 통해 검증된 설계로 구체화한다.

<HARD-GATE>
설계가 승인되기 전에는 어떤 구현 행위(코드 작성, 스캐폴딩, 파일 생성)도 금지한다.
</HARD-GATE>

### 1.1 규모 판단

프로젝트 컨텍스트 탐색 후 변경 규모를 판단한다:

- **Small** (단일 파일, ~50줄 이하): 질문 1-2개 → 설계 요약 → 승인 → Phase 3으로 직행
- **Standard** (여러 파일, 일반 기능): 전체 프로세스
- **Large** (멀티 서브시스템): 분해 → 서브프로젝트별 Standard 사이클

### 1.2 Standard 프로세스

1. **프로젝트 컨텍스트 탐색** — 파일, 문서, 최근 커밋 확인
2. **질문** — 한 번에 하나씩, 객관식 선호
   - 목적, 제약 조건, 성공 기준
   - 비기능 요구사항 (성능, 동시 사용자, 응답 시간)
   - 보안 요건 (인증/인가, 데이터 보호)
   - 기술 스택 검증 (기존 코드베이스 정합성)
   - 배포/인프라 제약
   - 모든 질문이 매번 필요한 것은 아님. 프로젝트 성격에 맞는 질문만 선별.
3. **접근법 제안** — 근거 기반 서치 후 제시
4. **MVP 범위 정의** — 핵심 기능 vs 후속 이터레이션 분리
5. **설계 제시** — 섹션별 복잡도에 맞게, 유저 승인
6. **스펙 문서 작성** — config의 `paths.specs`에 저장, 커밋
7. **스펙 셀프리뷰** — placeholder scan, 일관성, 모호성, 스코프 검사
8. **유저 리뷰** — 승인 후 Phase 2로 전환

### 1.3 접근법 제안 — 근거 기반 서치

**원칙: 재료는 검증된 것만, 요리(조합/판단)는 적극적으로**

- 근거 없는 추측 금지 ("~일 것 같다", "아마 ~")
- 조합과 판단은 적극적 — 수집한 근거를 바탕으로 프로젝트에 맞는 최적 방안 도출
- 판단의 근거를 반드시 함께 설명

**서치 소스 (병렬 실행):**

| 소스 | 도구 | 용도 |
|------|------|------|
| 공식 문서 | context7 | 라이브러리/프레임워크 API, 설정 |
| 실제 사례 | WebSearch | 베스트 프랙티스, 아키텍처 패턴 |
| 유사 구현체 | GitHub (WebSearch/WebFetch) | 스켈레톤 프로젝트, 참고 코드 |

**제안 수:** 명확한 최선이 있으면 1개+이유, 트레이드오프가 있으면 2-3개+추천.

### 1.4 후속 전환

| 규모 | 전환 대상 |
|------|----------|
| Small | Phase 3 (TDD) 직행 |
| Standard | Phase 2 (계획 수립) |
| Large | Phase 2 (서브프로젝트 단위) |

### Status 업데이트

Phase 1 완료 시: status yaml에 phase=1 completed, summary 기록. log에 Phase 1 완료 요약 append.

## Phase 2: 계획 수립

설계를 실행 가능한 계획으로 변환한다.

### 2.1 프로세스

1. **스코프 체크** — 멀티 서브시스템이면 계획 분리
2. **파일 구조 설계** — 신규/수정 파일 목록, 각 파일 책임 정의
3. **태스크 분해** — 2-5분 단위
   - 각 태스크: 테스트 명세 + 구현 힌트 (실제 코드 아님)
   - 태스크 간 의존성 그래프 + 병렬 가능 그룹 표시
4. **셀프리뷰** — 스펙 커버리지, placeholder scan, 타입 일관성
5. **유저 확인 게이트** — 계획 요약 제시 → 승인/수정/재작성
6. **계획 저장** — config의 `paths.plans`에 저장, 커밋
7. **실행 핸드오프** — 서브에이전트 or 인라인 실행 선택

### 2.2 태스크 구조

```markdown
### Task N: [컴포넌트명]

**Files:**
- Create: `src/auth/login.ts`
- Test: `src/auth/login.test.ts`

**테스트 설계:**
- 검증 대상: [행위 명세]
- 입력: [입력값]
- 기대 출력: [출력값]
- 엣지 케이스: [목록]
- 테스트 유형: 단위 / 통합 / E2E

**구현 힌트:**
- 인터페이스: [시그니처]
- 의존성: [라이브러리]

**의존성:** ← Task N (선행 태스크)
```

### 2.3 플레이스홀더 금지

절대 작성하지 않는다: "TBD", "TODO", "implement later", "Similar to Task N", 코드 없는 설명만 하는 스텝.

### 2.4 대규모 계획 — 단계적 확정

10개 이상 태스크: Phase A(핵심) → 유저 확인 → 구현 → Phase B(확장) → 유저 확인 → 구현.

### 2.5 유저 확인 게이트

```
📋 계획 요약
파일 구조: 신규 N개, 수정 N개
태스크 수: N개 (예상 커밋 N회)
의존성 그래프: [시각화]
스펙 커버리지: [매핑]
체크포인트 간격: (A) 매 태스크 (B) Phase 경계 ← 기본값 (C) 완료 후
진행할까요? [승인 / 수정 요청 / 재작성]
```

### Status 업데이트

Phase 2 완료 시: status에 phase=2 completed, tasks 목록, total_tasks 기록. log에 Phase 2 요약 append.

## Phase 3: TDD

Phase 2 테스트 명세를 실제 코드로 변환하며 RED→GREEN→REFACTOR 사이클로 구현한다.

### Iron Law

```
테스트 없이 작성된 프로덕션 코드는 삭제한다.
참고용으로 보지 않고, 적응시키지 않고, 삭제한다.
```

### 3.1 프레임워크 자동 감지

Phase 3 진입 시 프로젝트 설정 파일로 자동 감지. config의 `tdd.framework`이 auto가 아니면 그 값 사용.

```
package.json (jest) → npx jest / npx jest --coverage
package.json (vitest) → npx vitest run / npx vitest run --coverage
pyproject.toml → pytest / pytest --cov
Cargo.toml → cargo test / cargo tarpaulin
go.mod → go test ./... / go test -cover ./...
```

### 3.2 TDD 사이클

**RED** — 테스트 명세 기반으로 실패하는 테스트 작성. 실행하여 실패 확인 (MANDATORY).
**GREEN** — 테스트를 통과시키는 가장 단순한 코드. 실행하여 통과 확인 (MANDATORY).
**REFACTOR** — GREEN 이후에만. 체크: 긴 함수(>50줄), 깊은 중첩(>4레벨), 뮤테이션, 매직 넘버, 3회 이상 중복. 리팩터링 후 테스트 재실행 필수.
**COVERAGE** — config의 `defaults.coverage_threshold` 이상인지 확인. 미만이면 추가 TDD 사이클.

### 3.3 Spike — 탐색적 코딩

불확실한 기술일 때:
1. Spike 선언 (유저에게 알림)
2. 자유롭게 구현 (TDD 일시 해제)
3. 결과 정리
4. Spike 코드 삭제
5. TDD로 재구현

### 3.4 합리화 방어

| 변명 | 현실 |
|------|------|
| "너무 단순해서 테스트 불필요" | 단순한 코드도 깨진다. 30초면 된다. |
| "나중에 테스트 쓸게" | 나중에 쓴 테스트는 즉시 통과. 아무것도 증명 못함. |
| "이미 수동 테스트 했다" | 수동은 임시적. 기록 없고 재실행 불가. |
| "삭제하면 낭비" | 매몰 비용 오류. 신뢰할 수 없는 코드가 진짜 낭비. |
| "참고용으로만" | 보면 적응시킨다. 삭제는 삭제다. |
| "탐색이 필요하다" | Spike 경로를 선언하고 밟아라. |
| "TDD가 느리다" | 디버깅이 더 느리다. |

### 3.5 체크포인트

config의 `defaults.checkpoint_interval`에 따라 유저에게 진행 보고:
- 완료 태스크 / 전체 태스크, 현재 커버리지, 발견된 이슈, 다음 태스크

### 3.6 디버깅 연계

2회 이상 수정 실패 시 → Phase 4 진입.

### Status 업데이트

태스크 완료 시: status의 tasks[N].status=completed, task 번호 증가, coverage 업데이트.

## Phase 4: 디버깅 (필요 시에만)

추측이 아닌 증거 기반으로 근본 원인을 파악하고, 유저와 함께 해결 방향을 결정한다.

```
증거 없이 수정 금지. "아마 이것 때문일 것이다" → 금지. "일단 고쳐보자" → 금지.
```

### 진입 조건

- Phase 3에서 2회 이상 수정 실패
- 유저가 직접 버그 리포트
- 빌드/배포 실패

### 4.1 심각도 분기

**Trivial** (오타, import 누락): 원인 확인 → 수정 제안 → 유저 승인 → 적용 → 테스트 확인
**Standard** (일반 버그): 근본 원인 조사(자동) → 패턴 분석(자동) → 유저에게 보고 → 같이 방향 결정 → 수정 → 검증
**Complex** (멀티 컴포넌트, 재현 불가): 조사 시작 → 즉시 유저 보고 → 이후 전 과정 협업

### 4.2 표준 보고 템플릿

```markdown
## 디버깅 보고
**증상:** [관찰된 현상]
**조사 범위:** [확인한 파일, 로그, 컴포넌트]
**근본 원인 후보:**
1. [후보 A] — 확신도: 높음 — 증거: [...]
2. [후보 B] — 확신도: 중간 — 증거: [...]
**추천 해결 방향:** [설명]
**유저 결정 필요:** [방향 선택 / 추가 조사]
```

### 4.3 에스컬레이션

3회 실패 → "설계 변경이 필요할 수 있습니다" → (A) Phase 1 부분 회귀 (B) 우회 방안 (C) 계속 시도

### 4.4 TDD 연계

버그 발견 → 재현 테스트 작성 → 실패 확인 → 수정 → 통과 확인 → 회귀 테스트 보존

## Phase 5: 코드 리뷰

구현된 코드를 내부 병렬 리뷰 + Codex 외부 리뷰 2단계로 검증한다.

### 5.1 내부 리뷰

**Standard/Large (review_mode: parallel):** 3개 에이전트 병렬 실행
- Agent A — 코드 품질 (가독성, 패턴, 이뮤터블, 함수<50줄, 파일<800줄)
- Agent B — 보안 (시크릿, 입력 검증, SQL 인젝션, XSS, 인증/인가)
- Agent C — 테스트 (커버리지 80%+, mock 남용, 엣지 케이스, 격리성)

**Small 또는 review_mode: single:** 단일 code-reviewer

각 에이전트에는 세션 히스토리가 아닌 **git diff + 스펙 문서만** 전달.

### 5.2 리뷰 결과

```markdown
## 코드 리뷰 결과
### Critical (반드시 수정)
### Important (수정 추천)
### Minor (유저 판단)
```

유저 확인 → Critical 수정 필수, Important는 유저 선택, Minor는 유저 판단.

### 5.3 Codex 외부 리뷰

config의 `defaults.codex_review`가 true일 때만 실행. Small이면 생략.

```
/codex:review                → 일반 코드 리뷰
/codex:adversarial-review    → 더 강한 검증이 필요할 때
```

Codex 결과 → 유저에게 제시 → 동의하는 이슈 수정, 아닌 건 스킵 → Phase 6으로.

### Status 업데이트

Phase 5 완료 시: status에 phase=5 completed 기록. log에 리뷰 결과 요약 append.

## Phase 6: 검증

코드 리뷰를 통과한 코드가 실제로 동작하는지 검증한다.

```
"should work", "I'm confident" → 금지. 실행 결과를 보여줘야 한다.
```

### 6.1 검증 항목 자동 감지

프로젝트 설정 파일로 검증 커맨드 자동 구성. 감지 실패 시 유저에게 확인.

### 6.2 검증 레벨

- **FULL** (Standard/Large): 전체 테스트 + 빌드 + 린트 + 타입체크 + 커버리지
- **PARTIAL** (중간 변경): 관련 테스트 + 빌드 + 타입체크
- **QUICK** (Small/설정 변경): 빌드 통과만

### 6.3 5단계 게이트

각 검증 항목에 대해: IDENTIFY → RUN → READ → VERIFY → CLAIM.
CLAIM은 반드시 RUN의 실제 출력에 근거.

### 6.4 결과 보고

```markdown
## 검증 결과
✅ 테스트: N passed, 0 failed (커버리지 N%)
✅ 빌드: 성공
✅ 린트: 경고 0
✅ 타입체크: 에러 0
```

### 6.5 실패 시

Trivial: 수정 제안 → 유저 승인 → 재검증. Standard 이상: Phase 4 진입.

---

## Phase 7: 마무리

검증을 통과한 코드를 커밋하고 통합한다.

### 7.1 작업 완료 요약

```markdown
## 작업 완료 요약
**변경 내역:** 변경 파일 N개 (신규 N, 수정 N)
**품질:** 커버리지 N%, 리뷰 이슈 모두 해결, 검증 전체 통과
**커밋 이력:** [목록]
```

### 7.2 커밋

diff 요약 + 커밋 메시지 초안 제시 → 유저 승인 → 커밋. 메시지 형식: `<type>: <description>` (config의 `git.message_format`).

### 7.3 통합 방식 선택

```
(1) 로컬 머지 — base branch에 머지 (충돌 시 유저와 해결)
(2) PR 생성 — GitHub PR (제목 70자 이내, 본문 자동)
(3) 브랜치 유지 — 커밋만, 브랜치 유지
(4) 폐기 — "discard" 입력 필요
```

base branch 감지: config의 `git.base_branches` 순서대로 시도.

### 7.4 회고

```
"이번 워크플로우에서:
 - 불필요하거나 과했던 단계가 있었나요?
 - 부족했던 부분은요?"
```

→ log에 회고 내용 append → 필요 시 config에 반영

### 7.5 정리

- status/log를 `workflow-archive/`로 이동
- /clear 권유

## Status 관리

### 워크플로우 시작 시

`.claude/workflows/<branch-name>.yaml` 과 `<branch-name>-log.md` 생성.

### 업데이트 타이밍

| 이벤트 | status yaml | log md |
|--------|-------------|--------|
| Phase 전환 | phase 업데이트 | 완료 요약 append |
| 태스크 완료 | task, tasks[N] 업데이트 | - |
| 커밋 | last_updated 갱신 | 커밋 메시지 append |
| 에러/이슈 | notes에 추가 | 이슈 내용 append |
| 유저 결정 | 관련 필드 업데이트 | 결정 내용 append |
| 커버리지 | coverage 업데이트 | - |

### 워크플로우 완료 시

status/log → `.claude/workflow-archive/YYYY-MM-DD-<project>.*` 로 이동.
