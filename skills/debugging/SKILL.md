# Phase 4: 디버깅

추측이 아닌 증거 기반으로 근본 원인을 파악하고, 유저와 함께 해결 방향을 결정한다.

```
증거 없이 수정 금지. "아마 이것 때문일 것이다" → 금지. "일단 고쳐보자" → 금지.
```

## 진입 조건

- Phase 3에서 2회 이상 수정 실패 → 자동 진입
- 유저가 직접 버그 리포트 → 직접 진입
- 빌드/배포 실패 → 직접 진입

## 심각도 분기

**Trivial** (오타, import 누락): 원인 확인 → 수정 제안 → 유저 승인 → 적용 → 테스트 확인

**Standard** (일반 버그):
1. 근본 원인 조사 (자동) — 에러 메시지, 스택 트레이스, 관련 코드 분석
2. 패턴 분석 (자동) — 유사 패턴, 최근 변경사항, 영향 범위
3. 유저에게 보고 → 같이 방향 결정
4. 수정 → 검증

**Complex** (멀티 컴포넌트, 재현 불가):
1. 조사 시작
2. 즉시 유저 보고
3. **Agent Teams로 협업 디버깅** — 프론트/백엔드/DB 등 레이어별 에이전트가 서로 소통하며 원인 추적
4. 수정 및 검증

> Complex 디버깅은 에이전트 간 소통이 필요하므로 Subagent가 아닌 Agent Teams를 사용한다.

## 표준 보고 템플릿

```markdown
## 디버깅 보고
**증상:** [관찰된 현상]
**조사 범위:** [확인한 파일, 로그, 컴포넌트]
**근본 원인 후보:**
1. [후보 A] — 확신도: 높음 — 증거: [...]
2. [후보 B] — 확신도: 중간 — 증거: [...]
**추천 해결 방향:** [설명]
**유저 결정 필요:** [방향 선택 / 추가 조사]
```

## 에스컬레이션

3회 실패 → "설계 변경이 필요할 수 있습니다" → (A) Phase 1 부분 회귀 (B) 우회 방안 (C) 계속 시도

## TDD 연계

버그 발견 → 재현 테스트 작성 → 실패 확인 → 수정 → 통과 확인 → 회귀 테스트 보존

## Red Flags

| 생각 | 현실 |
|------|------|
| "아마 이게 원인일 거야" | 증거 없는 추측. 조사부터 해라. |
| "일단 고쳐보고 되나 보자" | 추측 수정은 새 버그를 만든다. |
| "이건 간단한 문제야" | 간단해 보이는 버그가 가장 위험하다. |
| "이전에 비슷한 거 고쳤으니까" | 비슷하게 보여도 원인이 다를 수 있다. |
