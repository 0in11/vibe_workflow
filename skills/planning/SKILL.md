# Phase 2: 계획 수립

설계를 실행 가능한 계획으로 변환한다. 제로 컨텍스트 엔지니어(또는 서브에이전트)도 따라할 수 있는 수준으로 구체적이어야 한다.

## 프로세스

1. **스코프 체크** — 멀티 서브시스템이면 계획 분리
2. **파일 구조 설계** — 신규/수정 파일 목록, 각 파일 책임 정의
3. **태스크 분해** — 2-5분 단위
   - 각 태스크: 테스트 명세 + 구현 힌트 (실제 코드 아님)
   - 태스크 간 의존성 그래프 + 병렬 가능 그룹 표시
4. **셀프리뷰** — 스펙 커버리지, placeholder scan, 타입 일관성
5. **유저 확인 게이트** — 계획 요약 제시 → 승인/수정/재작성
6. **계획 저장** — config의 `paths.plans`에 저장, 커밋
7. **실행 핸드오프** — 서브에이전트 or 인라인 실행 선택 → vibe:tdd 스킬로 전환

## 태스크 구조

```markdown
### Task N: [컴포넌트명]

**Files:**
- Create: `src/auth/login.ts`
- Test: `src/auth/login.test.ts`

**테스트 설계:**
- 검증 대상: [행위 명세]
- 입력: [입력값]
- 기대 출력: [출력값]
- 엣지 케이스: [목록]
- 테스트 유형: 단위 / 통합 / E2E

**구현 힌트:**
- 인터페이스: [시그니처]
- 의존성: [라이브러리]

**의존성:** ← Task N (선행 태스크)
```

## 의존성 그래프

태스크 간 의존 관계를 명시하여 병렬 실행과 크리티컬 패스를 식별한다. 병렬 가능한 태스크 그룹을 표시하여 Subagent 디스패치를 최적화한다.

## 플레이스홀더 금지

절대 작성하지 않는다: "TBD", "TODO", "implement later", "Similar to Task N", 코드 없는 설명만 하는 스텝.

## 대규모 계획 — 단계적 확정

10개 이상 태스크: Phase A(핵심) → 유저 확인 → 구현 → Phase B(확장) → 유저 확인 → 구현.

## 유저 확인 게이트

```
📋 계획 요약
파일 구조: 신규 N개, 수정 N개
태스크 수: N개 (예상 커밋 N회)
의존성 그래프: [시각화]
스펙 커버리지: [매핑]
체크포인트 간격: (A) 매 태스크 (B) Phase 경계 ← 기본값 (C) 완료 후
진행할까요? [승인 / 수정 요청 / 재작성]
```

## 실행 핸드오프

```
(1) Subagent 기반 (추천)
    — 태스크당 fresh subagent 디스패치
    — 병렬 가능 태스크는 동시 실행
    — 태스크 간 리뷰

(2) 인라인 실행
    — 현재 세션에서 순차 실행
    — 체크포인트에서 리뷰
```

## Status 업데이트

Phase 2 완료 시: status에 phase=2 completed, tasks 목록, total_tasks 기록. log에 Phase 2 요약 append.
